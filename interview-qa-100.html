Explain Terraform state — what is it, why does it matter, and what are the risks of state corruption?
AWSStripeShopify
+

Terraform state is a JSON file (terraform.tfstate) that maps your configuration to real-world infrastructure. It tracks resource IDs, metadata, and dependencies so Terraform knows what exists, what needs creating, and what needs destroying.

Why it matters: Without state, every plan would recreate all resources. State enables incremental changes, drift detection, and resource dependency resolution. Remote state (S3 + DynamoDB, Terraform Cloud) enables team collaboration with locking.

Risks of corruption:

  • Concurrent applies without locking → partial state writes
  • Manual resource deletion bypasses state → "resource exists in state but not in cloud"
  • State file deleted → Terraform treats all resources as new, causing duplication or conflicts
  • Sensitive data (passwords, keys) stored in plaintext in state — must use remote encrypted backends
💡 Senior answer: Always use remote state with DynamoDB locking on AWS. Use terraform state mv for refactoring, never manually edit state. Implement state file versioning and access control.

How do you structure Terraform modules for a large organization with 20+ teams and multiple environments?
GoogleNetflixLyft
+

Repository structure: Use a mono-repo for modules with separate repos for environments, or use Terraform Registry (private). Three-tier structure: modules/ (reusable), environments/ (dev/staging/prod), global/ (DNS, IAM, shared infra).

Module design principles:

  • Single responsibility — each module does one thing (e.g., a VPC module, not a "whole app" module)
  • Semantic versioning — pin module versions in environments (?ref=v2.1.0)
  • Input validation using validation blocks to fail fast
  • Output everything useful — downstream modules depend on outputs
  • Use Terragrunt for DRY environment configs, avoiding repeated backend and provider blocks
💡 Mention Atlantis or Terraform Cloud for PR-based workflow — automated plan on PR, apply on merge. This is what senior engineers implement at scale.

How do you handle infrastructure drift in Terraform, and what's your approach to importing existing resources?
MetaCloudflare
+

Drift detection: Run terraform plan in CI on a schedule (nightly). Tools like Driftctl or Terraform Cloud's drift detection surface changes made outside Terraform. Alert on drift so teams fix it rather than apply over it blindly.

Importing existing resources: Use terraform import <resource> <id> to bring existing infra under Terraform management. Terraform 1.5+ supports import blocks in config for repeatable, reviewable imports. After importing, run terraform plan to verify zero diff before proceeding.

⚠️ Common mistake: importing resources but not writing the matching HCL first — this leads to immediate destroy-and-recreate on next apply. Always write config, import, verify zero diff.
04
Terraform workspaces vs. directory-per-environment — which do you choose and why?
ShopifyAirbnb
+

Workspaces: Same code, different state files. Works well for identical environments (e.g., ephemeral dev environments). Problem: environments often diverge — prod may have extra resources, different sizes. Workspaces make this messy with conditionals everywhere.

Directory-per-environment (recommended at scale): Explicit, auditable, environments can diverge cleanly. Use Terragrunt to avoid DRY violations — share module sources but override variables per environment. Each directory has its own state, reducing blast radius.

Rule of thumb: Use workspaces for truly identical ephemeral environments. Use directories for long-lived environments that will inevitably diverge.

When would you choose Pulumi over Terraform, and what are the trade-offs?
StartupsMicrosoft
+

Choose Pulumi when: Your team is software-engineering-heavy and wants loops, functions, classes, unit tests for infra. Dynamic resource creation (create N resources based on a list) is trivial in Python/TypeScript. Better for complex logic, testing, and code reuse.

Choose Terraform when: Team is ops-heavy, HCL is more approachable, existing ecosystem (modules, providers) is mature and battle-tested. Better tooling support (Atlantis, Spacelift, Terraform Cloud).

Trade-offs: Pulumi has a steeper learning curve for ops folks; Terraform has a learning curve for devs. Both use state files with similar risks. Pulumi's policy-as-code (CrossGuard) is powerful but less mature than Sentinel.

StripeSquarePlaid
+

  • Rotate credentials used by Terraform's AWS provider via OIDC (no static keys)

Your Terraform plan takes 20 minutes for a large codebase. How do you optimize it?
UberLyft
+

Break up state: Split monolithic state into smaller, domain-scoped states (network, compute, data, apps). Each state is faster to plan independently. Use terraform_remote_state or Terragrunt to share outputs between states.

Use -target selectively: For urgent changes, target specific resources. Not for routine use as it risks missing dependent changes.

Parallelism: Increase -parallelism flag (default 10) for read-heavy plans against large infra. Use terraform plan -refresh=false when you've just applied and state is fresh. Provider-level caching for repeated API calls.

HashicorpConfluent
+

  • terraform state mv — rename resources in state (e.g., when moving to a module)
  • moved blocks (Terraform 1.1+) — declarative refactoring, reviewable in code, automatically handles state moves on next apply

Explain the difference between node affinity, pod affinity, taints, and tolerations — when would you use each?
GoogleAmazon
+

Node Affinity: Attracts pods to nodes with specific labels (e.g., run ML workloads on GPU nodes). Replaces nodeSelector with more expressive rules. Use requiredDuringScheduling for hard rules, preferredDuringScheduling for soft rules.

Pod Affinity/Anti-affinity: Co-locate or spread pods relative to other pods. Anti-affinity ensures replicas spread across nodes/zones for HA. Affinity co-locates related services to reduce latency.

Taints & Tolerations: Taints repel pods from nodes (kubectl taint nodes gpu-node key=value:NoSchedule). Tolerations allow pods to override taints. Used to reserve specialized nodes (GPU, high-memory) for specific workloads only.

💡 Real use case: Taint GPU nodes with nvidia.com/gpu:NoSchedule. Only ML training pods with the matching toleration can schedule there, preventing general workloads from consuming expensive GPU capacity.
12
Compare HPA, VPA, and KEDA — when would you use each scaling approach?
NetflixUber
+

HPA (Horizontal Pod Autoscaler): Scales pod count based on CPU/memory or custom metrics. Best for stateless services with predictable load patterns. Reacts to current metric values — scale out when CPU > 70%.

VPA (Vertical Pod Autoscaler): Adjusts CPU/memory requests of existing pods. Requires pod restart (in-place VPA is in beta). Use for right-sizing, not reactive scaling. Best combined with HPA on different metrics.

KEDA (event-driven): Scales to zero and up based on external event sources — Kafka lag, SQS queue depth, Prometheus query, HTTP requests. Game-changer for async workloads. If queue is empty → 0 pods; queue fills → scale up instantly.

💡 Use KEDA + HPA together: KEDA handles scale-to-zero and event-driven scaling, HPA handles CPU-based scaling for HTTP services. Don't put VPA on the same pods as HPA targeting the same metric.

Explain Kubernetes networking — how does pod-to-pod communication work, and what's the advantage of Cilium over Calico?
CloudflareDatadog
+

K8s networking model: Every pod gets a unique IP. Pods can communicate directly (no NAT) using the CNI plugin. Services provide stable virtual IPs (ClusterIP) with kube-proxy load balancing via iptables or IPVS.

Calico: BGP-based routing, mature, widely adopted. Network policies enforced via iptables (older approach). Works well at moderate scale.

Cilium (eBPF-based): Uses eBPF in the Linux kernel — bypasses iptables entirely, significantly lower latency and CPU overhead at scale. Provides deep observability (Hubble), L7 network policies, transparent encryption, service mesh capabilities without sidecars. At 10k+ pods, Cilium's eBPF approach is dramatically more performant than iptables-based CNIs.

What is the Kubernetes Operator pattern, and when should you write your own vs. use an existing one?
Red HatVMware
+

An Operator is a controller that extends Kubernetes with domain-specific knowledge via Custom Resource Definitions (CRDs). It watches for CRD events and runs reconciliation logic to drive actual state toward desired state — like a human operator encoded in software.

Use existing when: Managing databases (PostgreSQL Operator, MongoDB Operator), Kafka (Strimzi), Prometheus, Cert-Manager. These are battle-tested and cover 90% of use cases.

Write your own when: You have proprietary application logic for Day 2 operations (backup, failover, config updates) that no existing operator covers. Use Kubebuilder or Operator SDK (Go). The reconciler loop implements your business logic.

15
How do you ensure zero downtime during Kubernetes rolling updates for a stateful service?
StripeSquare
+

Rolling update config: Set maxUnavailable: 0 and maxSurge: 1 to always have full capacity during rollout. For stateful sets, this is more complex — pods are updated in reverse ordinal order.

Pod Disruption Budgets (PDB): Define minimum available replicas during voluntary disruptions (node drains, cluster upgrades). Prevents cluster operations from taking down too many pods simultaneously.

Readiness probes: The new pod must pass readiness checks before old pod is terminated. Configure both liveness and readiness probes correctly — readiness controls traffic; liveness controls restarts.

Graceful shutdown: Set terminationGracePeriodSeconds appropriately. Handle SIGTERM in the application to drain in-flight requests before exiting. Use preStop lifecycle hook with a sleep to allow time for endpoints to be removed from kube-proxy.

A pod keeps getting OOMKilled. How do you diagnose and fix it without just increasing memory limits?
AmazonUber
+

Diagnosis: kubectl describe pod (check OOMKilled exit code 137), kubectl top pods for current usage, check memory usage trends in Grafana/Prometheus. Use kubectl exec + heap profiling tools for the specific runtime (jmap for JVM, pprof for Go).

Fix the root cause first:

  • Memory leak in application code — profile and fix the leak
  • JVM heap not configured for containers — set -XX:MaxRAMPercentage=75 instead of absolute heap values
  • Memory requests too low → node overcommits → pod gets killed under contention; raise requests to match actual usage
  • Caching too aggressively — cap cache sizes in the app

If usage is legitimate, then set limits equal to requests (Guaranteed QoS class — highest priority, last to be killed). Use VPA to right-size based on historical data.

How do you back up and restore a Kubernetes cluster, and what's your DR strategy?
CKA ExamEnterprise
+

etcd backup: etcdctl snapshot save for control plane state. Automate snapshots to S3/GCS on a schedule. Test restores regularly — a backup you haven't restored is just a guess.

Application-level backup: Velero for backing up K8s resources and persistent volumes. Schedules, retention policies, cross-region copies. For stateful apps, use application-aware backup (database dumps, not just PVC snapshots).

DR strategy: Define RTO/RPO per service tier. Active-active across regions (highest cost, near-zero RTO) vs. active-passive with automated failover (minutes RTO) vs. backup-restore (hours RTO). GitOps dramatically reduces recovery time — cluster state is in Git, redeploy is fast.

You're running 5,000 pods and the scheduler is slow. What are the performance bottlenecks in Kubernetes at scale?
MetaByteDance
+

API server: etcd is the bottleneck at scale — use SSD, tune etcd heartbeat and election timeout, run dedicated etcd nodes. Increase API server rate limits. Cache heavy list/watch operations.

Scheduler: Use scheduler profiles to separate workloads (GPU jobs, batch, real-time). Descheduler for rebalancing. Consider scheduler extenders or custom schedulers for specialized workloads.

kube-proxy: Switch from iptables to IPVS mode for O(1) routing at scale (iptables is O(n)). Or use Cilium eBPF (no kube-proxy at all).

Node count: Each node increases API server watch overhead. Use larger nodes (fewer nodes) to reduce control plane pressure. EKS/GKE managed control planes auto-scale for you.

How do you perform a zero-downtime Kubernetes version upgrade on a production cluster with 200 nodes?
ShopifyGitHub
+

Pre-upgrade: Check API deprecations (removed APIs in target version), test in staging first, back up etcd, review release notes for breaking changes. Use pluto tool to scan for deprecated APIs in your manifests.

Control plane: Upgrade control plane first (API server, controller manager, scheduler). On managed K8s (EKS/GKE), this is one click with minimal downtime.

Node upgrade (blue-green approach): Provision new node group with new K8s version → gradually cordon and drain old nodes (respecting PDBs) → workloads migrate to new nodes → verify → delete old node group. No downtime, easy rollback by keeping old nodes.

What problems does a service mesh solve, and when is it not worth the overhead?
LyftAirbnb
+

Problems solved: mTLS between all services (zero-trust networking), traffic management (canary, weighted routing, retries, timeouts, circuit breaking), observability (automatic L7 tracing, metrics per service-to-service path), fault injection for chaos testing — all without application code changes.

When it's not worth it: Small teams (<5 services), monolithic architectures, teams without the expertise to operate Istio's complexity, latency-sensitive applications (sidecar adds ~1-5ms per hop). Cilium's eBPF approach gives similar benefits with less overhead — consider it as an alternative.

⚠️ Istio has a steep learning curve and can cause outages if misconfigured. Don't adopt it just because it's cool — adopt it when you have specific problems it solves.

What are the four GitOps principles, and how does ArgoCD differ from Flux v2?
WeaveworksCodefresh
+

GitOps principles (OpenGitOps): 1) Declarative — system state described declaratively. 2) Versioned & immutable — desired state stored in Git as source of truth. 3) Pulled automatically — software agents pull from Git (not pushed to). 4) Continuously reconciled — agents compare actual vs. desired and remediate drift.

ArgoCD vs Flux: ArgoCD has a rich UI, RBAC, SSO, ApplicationSets for multi-cluster — better for large teams needing governance. Flux is more Kubernetes-native (everything is a CRD), lighter weight, better for GitOps purists and automated pipelines. Both support Helm, Kustomize, and plain YAML. ArgoCD is more popular in enterprises; Flux in platform-engineering-heavy orgs.

Explain canary deployment vs. blue-green vs. feature flags — when would you use each?
NetflixFacebook
+

Canary: Gradually shift traffic to new version (5% → 20% → 50% → 100%). Automated analysis (error rate, latency) decides to promote or rollback. Low risk, production-validated. Use for backend services. Argo Rollouts + Prometheus metrics automate this.

Blue-Green: Run two identical environments; switch traffic instantaneously. Zero-downtime cutover, instant rollback (just point back). Higher cost (2x infra). Use for major version changes or database schema migrations where you need clean cutover.

Feature Flags: Deploy code to all users but enable features for a subset. No deployment needed to change behavior. Use for A/B testing, gradual rollouts to user segments, kill switches. LaunchDarkly, OpenFeature are popular. Separates deployment from release.

How do you secure a CI/CD pipeline against supply chain attacks? Explain SLSA and Sigstore.
GoogleGitHub
+

Supply chain threats: Malicious dependencies (SolarWinds-style), compromised CI runners, tampered artifacts, unsigned images pulled in production.

SLSA (Supply chain Levels for Software Artifacts): Framework with 4 levels of supply chain integrity. Level 1: basic provenance. Level 2: hosted build service. Level 3: hardened build — build artifacts are signed, build environment is audited. SLSA provenance attests "this artifact was built by this pipeline from this source."

Sigstore / Cosign: Free, keyless signing of container images using ephemeral keys tied to OIDC identity. Images signed at build time; Kubernetes (via Policy Controller) verifies signatures before pulling. Creates an immutable transparency log (Rekor) — any tampering is detectable.

GoogleEtsy

What are DORA metrics and how do you use them to improve your engineering team's delivery performance?
MetaLinkedIn
+

Four DORA metrics: 1) Deployment Frequency — how often you deploy to prod (elite: multiple/day). 2) Lead Time for Changes — commit to prod time (elite: <1 hour). 3) Change Failure Rate — % of deployments causing incidents (elite: 0-15%). 4) Time to Restore Service — how fast you recover (elite: <1 hour).

Using them: High deployment frequency with high failure rate → improve testing, add canary. Low deployment frequency → reduce batch size, improve CI speed. High lead time → identify bottlenecks (long test suites, manual approvals, code review queue). Measure trend over time, not absolute values.

How do you use ArgoCD ApplicationSets to manage deployments across 10 clusters and 5 environments?
SpotifyZalando
+

ApplicationSets are a template engine for ArgoCD Applications. Define one ApplicationSet and it generates multiple Applications automatically based on generators.

Generators: Cluster generator (create an App per cluster), List generator (fixed list of envs), Matrix generator (product of clusters × environments), Git generator (one App per directory in a repo — great for microservices). Combine them to deploy service X to all prod clusters but not dev.

How do you manage secrets in a GitOps workflow without storing sensitive values in Git?
HashiCorpStripe
+

External Secrets Operator (ESO): Sync secrets from AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager into Kubernetes Secrets. You store a reference in Git (not the secret), ESO pulls the actual value at runtime. Best practice for GitOps.

Sealed Secrets: Encrypt secrets with a cluster-specific public key. Encrypted values safe to commit to Git. Only the cluster can decrypt. Simpler to operate than ESO but no dynamic rotation.

💡 For enterprise: ESO with Vault + dynamic secrets (Vault generates short-lived DB credentials per pod) is the most secure architecture. No long-lived credentials ever exist in Kubernetes.

What is Platform Engineering, and how does it differ from traditional DevOps? How do you measure an IDP's success?
SpotifyAirbnb
+

Platform Engineering treats the internal infrastructure platform as a product, with developers as customers. Instead of embedding a DevOps engineer in every team (traditional), a central platform team builds self-service capabilities — golden paths, templates, CLI tools — that enable any developer to deploy, monitor, and operate their service without deep infra knowledge.

How does OPA Gatekeeper work? Write an example policy that prevents containers from running as root.
GoogleEnterprise
+

OPA Gatekeeper is an admission controller webhook. When a resource is created/updated, Gatekeeper evaluates it against ConstraintTemplates (written in Rego). If the policy is violated, the request is denied with a descriptive error.

ConstraintTemplate example:

Define the policy logic in Rego: deny if input.review.object.spec.containers[_].securityContext.runAsNonRoot != true. Create a Constraint resource that activates this template for all pods in specific namespaces.

Kyverno alternative: Kyverno uses YAML instead of Rego — more accessible for ops teams. Same admission webhook approach but with mutation capability (auto-add labels, default values). For most teams, Kyverno is easier to maintain; OPA has more expressive power for complex policies.

Explain the Zero Trust security model and how you implement it in a cloud-native environment.
Google BeyondCorpMicrosoft
+

Zero Trust: "Never trust, always verify." No implicit trust based on network location (inside the perimeter). Every request is authenticated and authorized regardless of where it originates.

Implementation:

  • Identity-first: Every service has a cryptographic identity (SPIFFE/SPIRE issues X.509 SVIDs). mTLS between all services using these identities.
  • Micro-segmentation: Network policies restrict pod-to-pod traffic to only what's necessary. Default-deny all, explicit allow.
  • Short-lived credentials: No long-lived API keys. OIDC, workload identity federation, Vault dynamic secrets.
  • Continuous verification: Authorization at every request (OPA), not just at login. Re-verify on each API call.

How do you design an on-call rotation and alerting strategy that minimizes alert fatigue while maintaining reliability?
Google SRENetflix
+

Alert design principles: Every alert must be actionable (if you can't do anything at 2am, it shouldn't page). Alerts should be symptoms, not causes (page on "user can't checkout" not "CPU > 80%"). Explain the self-attention mechanism in transformers — what are Q, K, V and why does it scale better than RNNs?
Google DeepMindOpenAI
+

Self-attention computes relationships between all tokens in a sequence simultaneously. For each token, we compute Query (what am I looking for?), Key (what do I contain?), and Value (what information do I provide?).

Attention score = softmax(Q·Kᵀ / √d_k) · V. The score between Q and K determines how much attention to pay to each token. The output is a weighted sum of Values. Multi-head attention runs this in parallel multiple times with different weight matrices, capturing different relationship types.

Why better than RNN: RNNs process tokens sequentially — information from token 1 must pass through tokens 2,3,4...N to influence token N. Long-range dependencies degrade. Attention directly computes token 1 ↔ token N relationship in one step. Also fully parallelizable (RNNs are sequential, slow to train).

Why do LLMs struggle with very long context windows even when supported, and what strategies address this?

+

"Lost in the middle" problem: Research shows LLMs recall information better when it's at the beginning or end of the context window. Information in the middle is often ignored even if technically within the context limit. Attention becomes diffuse at very long sequences.

Computational cost: Attention is O(n²) in sequence length — 4x longer sequence = 16x more compute. KV cache grows linearly, hitting GPU memory limits.

Strategies:

  • RAG — retrieve only relevant chunks instead of stuffing everything
  • Hierarchical summarization — compress long documents before feeding
  • Reranking — put most relevant content at start and end (not middle)
  • Chunking with overlap — process in windows with context preservation
  • Contextual compression — extract relevant sentences, not full docs

Explain temperature, top-p, and top-k sampling. When would you use temperature=0 vs temperature=1?
AI Product interviews
+

Temperature: Controls distribution sharpness. Temperature=0 → always picks highest probability token (greedy, deterministic). Temperature=1 → samples from raw probability distribution (creative, varied). Temperature=2 → very random, often incoherent.

Top-k: Restrict sampling to top-k tokens by probability. Prevents sampling very unlikely tokens regardless of temperature.

Top-p (nucleus sampling): Sample from the smallest set of tokens whose cumulative probability ≥ p. More adaptive than top-k — uses more options when distribution is flat, fewer when one option dominates.

When to use: Temperature=0 for factual Q&A, code generation (need correctness), structured output extraction. Temperature=0.7-1 for creative writing, brainstorming, diverse outputs. Avoid temperature=0 for tasks needing variety; avoid high temperature for precise tasks.

Why do LLMs hallucinate, and what are the technical approaches to reducing hallucinations in production?
All AI companies
+

Root causes: LLMs are trained to produce plausible-sounding text, not necessarily true text. They interpolate/extrapolate training data without a ground-truth verifier. They have no mechanism to say "I don't know" — they always generate something. The model conflates high confidence in fluency with high confidence in facts.

Technical mitigations:

  • RAG: Ground responses in retrieved documents — model generates from evidence, not memory. Dramatically reduces factual hallucinations.
  • Constrained generation: Force JSON output, use function calling — structure reduces hallucination space
  • Calibration: Instruct model to express uncertainty ("I'm not certain, but…"). RLHF fine-tuning can teach models to refuse rather than hallucinate.
  • Self-consistency: Sample multiple responses, take the majority answer
  • Verification step: Use a second LLM to verify claims against source material

Explain RLHF — how does it work, and how does DPO simplify the process?

+

How do you evaluate a RAG system? Explain the RAGAS metrics and how they guide improvements.

+

RAGAS metrics:

  • Faithfulness: Is the answer grounded in the retrieved context? (Hallucination metric). Low faithfulness = model is making up info not in context.
  • Answer Relevance: Does the answer actually address the question? Low = model answers something different.
  • Context Recall: Does retrieved context contain the necessary information? Low = retrieval is failing, missing relevant docs.
  • Context Precision: Is the retrieved context focused on the question? Low = too much noise, irrelevant chunks retrieved.

Diagnostic use: Low faithfulness → fix generation (better prompt, smaller context window). Low context recall → fix retrieval (better chunking, hybrid search, more chunks). Low context precision → add reranking, metadata filtering. Low answer relevance → fix prompt, check if question is unanswerable from corpus.

Explain the ReAct pattern for AI agents. What are the failure modes and how do you make agents reliable?
Agentic AI companies
+

ReAct (Reasoning + Acting): Interleaves reasoning steps with action steps. Agent thinks ("I need to look up X to answer this"), takes action (tool call), observes result, thinks again ("Now I know X, I need Y"), acts again... until task complete. More reliable than pure chain-of-thought because actions provide grounding feedback.

Failure modes:

  • Infinite loops — agent loops tool calls without progress. Fix: max step limit.
  • Wrong tool selection — agent picks the wrong tool. Fix: better tool descriptions, structured tool selection.
  • Cascading errors — early mistake compounds. Fix: verification checkpoints.
  • Context window overflow — long reasoning chains exceed context. Fix: summarize history, prune old messages.
  • Hallucinated tool outputs — agent pretends a tool call returned useful info. Fix: enforce real tool execution, validate outputs.

Explain LoRA mathematically and why it enables fine-tuning large models on consumer GPUs.
ML research companies
+

Fine-tuning updates W (weight matrix) as W' = W + ΔW. ΔW has the same dimensions as W — for a 4096×4096 weight matrix in Llama 3, that's 16M parameters per layer.

LoRA insight: ΔW has low intrinsic rank. Instead of training ΔW directly, decompose it: ΔW = B·A where A is r×d and B is d×r, with rank r ≪ d (e.g., r=16 vs. d=4096). Now you're training r×d + d×r = 2×16×4096 = 131K params instead of 16M — a 120x reduction.

QLoRA adds quantization: Quantize frozen base model to 4-bit (NF4), load LoRA adapters in 16-bit. 70B model fits in 48GB GPU (2×A100) instead of requiring 140GB. Makes fine-tuning accessible. Adapters can be merged back into the base model for inference or served separately.

Best practices for designing tools/functions for LLM agents — what makes a tool easy for a model to use correctly?

+

What's the difference between SFT, DPO, and ORPO? When would you use each in a fine-tuning pipeline?
AI research labs
+

SFT (Supervised Fine-tuning): Train on demonstration data (instruction → good response). Teaches the task/format/style. Required first step. Uses standard cross-entropy loss. Simple, stable, well-understood.

DPO (Direct Preference Optimization): After SFT, train on preference pairs (chosen response > rejected response) using a contrastive loss. Aligns model behavior toward preferred outputs. Simpler than RLHF (no separate reward model). Now the standard alignment technique.

ORPO (Odds Ratio Preference Optimization): Combines SFT and preference optimization in a single training pass. No separate SFT phase needed. Faster, cheaper, less data. Shows competitive results to SFT+DPO pipeline. Increasingly popular in 2026 for resource-constrained fine-tuning.

Typical pipeline: Pre-trained model → SFT on task data → DPO on human preferences → deployment.

How do you prevent catastrophic forgetting and overfitting when fine-tuning an LLM on a small domain dataset?
AI fine-tuning
+

Catastrophic forgetting: Fine-tuning on domain data overwrites general capabilities. LoRA mitigates this significantly because only adapters are trained, base weights unchanged. Keep learning rate small (1e-4 to 5e-5). Mix domain data with general instruction data (70/30 ratio).

Preventing overfitting on small datasets:

  • Early stopping — monitor validation loss, stop when it starts increasing
  • Fewer epochs — 1-3 epochs often sufficient for instruction tuning. More = overfitting.
  • Data augmentation — paraphrase training examples to create variety
  • Lower rank LoRA — smaller r = less capacity = less overfitting risk
  • Dropout on LoRA layers

Evaluation: Track both domain benchmark (should improve) and general benchmarks like MT-Bench (should not degrade). If general capability drops, you have catastrophic forgetting.

How does vLLM's PagedAttention work, and why does it dramatically increase LLM serving throughput?
LLM servingBerkeley AI
+

KV Cache problem: During autoregressive generation, LLMs cache key-value tensors for all previous tokens (KV cache). Traditional systems pre-allocate max sequence length of GPU memory per request — enormous waste when requests are shorter, limits parallel requests dramatically.

PagedAttention: Inspired by OS virtual memory paging. KV cache stored in non-contiguous pages (like OS pages). Pages allocated dynamically as needed. Different requests can share KV cache memory pages (e.g., prefix caching — all requests with same system prompt share those KV pages). Memory utilization improves from ~30% to near 100%.

Continuous batching: Instead of waiting for all requests in a batch to finish before starting new ones, vLLM slots in new requests as others complete. Eliminates GPU idle time. Combined with PagedAttention: 24x throughput improvement over naive implementations in benchmarks.

What are the key latency metrics for LLM APIs, and how do you optimize for each?
Production AI
+

TTFT (Time to First Token): How long until user sees the first word. Most important for UX — streaming means users see response immediately. Optimize: reduce prompt length, use speculative decoding, model parallel across GPUs, prompt caching for repeated prefixes.

TPS (Tokens Per Second) / throughput: Generation speed. Optimize: larger batch size, tensor parallelism (multi-GPU), quantization (AWQ 4-bit → 2-4x faster), speculative decoding with draft model.

Total latency: TTFT + (output_tokens / TPS). For chat, TTFT matters most. For batch processing, TPS matters most.

Speculative decoding: Small fast draft model generates N tokens speculatively. Large model verifies all N in parallel (one forward pass). Accept correct tokens, reject wrong ones. 2-3x speedup for typical chat outputs where the distribution is peaked.
