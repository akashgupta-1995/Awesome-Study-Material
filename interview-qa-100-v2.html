You have 30 VPCs across 3 AWS accounts. How do you design the network architecture to minimize cost and complexity?
AWSNetflix
+

Hub-and-spoke with AWS Transit Gateway: Single TGW acts as the central router. All VPCs attach to it. No VPC peering mesh (which becomes O(n²) connections). TGW route tables segment traffic — prod VPCs can't reach dev by default.

Account structure: Network account owns TGW and shared services (DNS, inspection). Spoke accounts attach. AWS RAM (Resource Access Manager) shares TGW across accounts without complex cross-account IAM.

Cost optimization: Minimize cross-AZ traffic (expensive). Centralize egress via a shared NAT Gateway or VPC Endpoint for S3/DynamoDB (avoids NAT charges). Use VPC Lattice for service-to-service communication — pay per request, not per connection.

💡 Security tip: Add an inspection VPC in the hub with AWS Network Firewall or a third-party NGFW for all east-west traffic inspection without hairpinning through the internet.
02
Explain split-horizon DNS and how you implement it in a hybrid cloud environment where on-prem and cloud coexist.
EnterpriseHybrid Cloud
+

Split-horizon DNS returns different answers for the same hostname depending on the querying network. api.company.com resolves to a private IP for internal clients and a public IP/CDN for internet clients.

Implementation: Route 53 Private Hosted Zones serve private IPs to clients inside VPCs. Route 53 Public Hosted Zones serve public IPs. Route 53 Resolver endpoints enable on-premises DNS servers to forward queries to Route 53 and vice versa — enabling both directions of hybrid resolution.

On-prem integration: Inbound Resolver endpoint (on-prem → cloud). Outbound Resolver endpoint (cloud → on-prem for legacy DNS). Conditional forwarding rules direct specific domains to the right DNS server. This eliminates the need to maintain parallel DNS databases.

03
A CTO asks you to "eliminate cloud vendor lock-in." What's your honest assessment and recommended strategy?
ArchitectureStrategy
+

Honest reality: True multi-cloud portability is extremely expensive and often counterproductive. You sacrifice managed services (RDS, BigQuery, Pub/Sub) for portable but lower-value alternatives. The engineering cost of maintaining portability layers typically exceeds the risk of vendor lock-in.

Smart approach — portability where it matters:

  • Containerize workloads (K8s + Helm) — can move clusters, even if you don't
  • Abstract the compute layer, use cloud-native storage/database where justified
  • Keep data ownership clean — avoid proprietary data formats, use open standards
  • Multi-cloud for specific reasons: resilience (active-active DR), best-of-breed (GCP for ML, AWS for breadth), compliance (data residency)

The right answer is "portability where valuable, cloud-native where optimal" — not blanket abstraction.

Design a global CDN strategy for an API serving 50M requests/day across APAC, EU, and US regions.
ShopifyCloudflare
+

Edge caching for APIs: Not all API responses are cacheable, but many are. Identify GET endpoints with stable data (product catalog, config, reference data) and cache at edge with appropriate TTLs. CloudFront or Cloudflare Workers with cache rules. Use Vary headers carefully — each Vary dimension multiplies cache space.

Active origins per region: Route users to nearest origin via latency-based DNS (Route 53) or Anycast (Cloudflare). Origin in each of US/EU/APAC. Replication strategy depends on data consistency requirements.

Cache invalidation: Surrogate keys (Cloudflare Cache Tags) allow targeted invalidation — when product X changes, invalidate all edges caching X. Stale-while-revalidate for non-critical data: serve stale immediately, refresh in background. Result: global p50 latency <50ms for cached, <200ms for cache miss.

How do you implement least-privilege IAM at scale across 50 AWS accounts using AWS Organizations?
AWSEnterprise
+

Service Control Policies (SCPs): At the Organization level, SCPs are guardrails — they can only restrict, never grant. Deny specific regions (data residency), deny disabling CloudTrail, deny creating internet gateways in production accounts. These apply to every principal including root. Cannot be overridden.

Permission Boundaries: IAM administrators can create roles, but only within the permission boundary. Prevents privilege escalation — a developer with IAM access can't create a role with more permissions than themselves.

IAM Identity Center (SSO): Single identity source, Permission Sets define what users can do per account. Temporary credentials, no long-lived access keys for humans. Access Analyzer continuously audits unused permissions and suggests least-privilege refinements based on actual usage (CloudTrail data).

Your AWS bill shows $80k/month in data transfer charges. How do you systematically reduce this without impacting performance?
AWS FinOps
+

Identify the sources first: AWS Cost Explorer → Data Transfer → break by type (inter-AZ, egress, CloudFront). Typically: inter-AZ transfer (often invisible but expensive), S3 egress, NAT Gateway processing, cross-region replication.

Inter-AZ reduction: Co-locate services in same AZ for traffic-heavy pairs (stateful connections). Topology-aware load balancing in K8s (keep traffic in zone). This alone can save 30-50% on data transfer.

S3/DynamoDB VPC endpoints: Traffic to S3 through NAT Gateway costs $0.045/GB. VPC endpoints for S3 and DynamoDB are free — traffic stays on AWS backbone. Significant for data-heavy workloads.

CloudFront: Move static content and API caching to CloudFront — egress from CloudFront is cheaper than from EC2/ALB. Often 50-80% cheaper for cacheable content. Use S3 Origin Shield to reduce origin requests.

What's the operational overhead of running Istio at scale, and when would you switch to Linkerd or Cilium instead?
Service meshPlatform Eng
+

Istio overhead reality: Envoy sidecar adds 50-200MB RAM per pod. At 500 pods, that's 25-100GB extra RAM cluster-wide. CPU overhead ~5-15% per hop. Control plane (istiod) needs resources. Complex CRD surface area — VirtualService, DestinationRule, PeerAuthentication — steep learning curve, easy to misconfigure causing outages.

Switch to Linkerd when: You need mTLS and observability but not Istio's full feature set. Linkerd is Rust-based, uses micro-proxies (~10MB each), dramatically lower overhead, simpler operation. Less feature-rich but more operationally predictable. Ideal for teams that don't need Istio's advanced traffic management.

Switch to Cilium (eBPF) when: You want service mesh capabilities without sidecars at all. Cilium Mesh uses eBPF in kernel — zero additional pods, lowest latency overhead, excellent observability via Hubble. Best choice for greenfield 2026 deployments focused on performance.

What are the trade-offs of running databases inside Kubernetes vs. using managed cloud databases in 2026?
ArchitectureStorage
+

Running databases in K8s (with operators): Better resource utilization, co-location with app code, consistent GitOps deployment model, no cloud-specific lock-in. Mature operators exist (CloudNativePG for Postgres, Vitess for MySQL sharding, Strimzi for Kafka). Required for cost optimization at scale and for multi-cloud portability.

Challenges: CSI storage with proper IOPS provisioning (EBS gp3 with provisioned IOPS). Node anti-affinity to spread replicas across AZs. Careful PDB configuration. Backup coordination with the operator. You need deep database expertise — the operator automates operations but doesn't replace understanding.

Managed databases win when: Team lacks database expertise. Need HA without complexity. Compliance requirements for managed service (some regulated industries). Early stage where operational simplicity matters more than cost.

💡 2026 trend: Large companies like Shopify run thousands of MySQL clusters in Kubernetes. The operators are mature enough that this is mainstream for organizations with the expertise.

How do you implement multi-tenancy in Kubernetes for a SaaS platform where tenants must be strictly isolated?
SaaSMulti-tenant
+

Namespace-per-tenant (soft isolation): RBAC limits access per namespace. ResourceQuotas prevent one tenant consuming all cluster resources. NetworkPolicies enforce namespace isolation. Suitable for internal teams or trusted customers. Not suitable for adversarial tenants — namespaces share the kernel.

Cluster-per-tenant (hard isolation): Complete isolation at the Kubernetes API and infrastructure level. More expensive, more complex to manage (Cluster API + ArgoCD ApplicationSets handles this at scale). Required for high-security or compliance-sensitive tenants.

Virtual clusters (vcluster): Lightweight K8s control planes running inside namespaces. Each tenant gets their own API server, scheduler, but shares worker nodes. Best of both: isolated control plane, shared compute cost. Strong isolation model gaining popularity in 2026 for developer environments and SaaS.

How does Karpenter improve on the Cluster Autoscaler, and what's your strategy for maximizing spot instance utilization?
AWS EKSCost
+

Cluster Autoscaler limitations: Tied to fixed node groups. Slow to provision (polls every 10 seconds). Can't consider the specific pod requirements when selecting instance type — uses pre-defined groups. Consolidation is basic.

Karpenter advantages: Watches pending pods directly, provisions the optimal instance type for those specific requirements in seconds. Supports hundreds of instance types — Karpenter picks the cheapest that fits. Bin-packing aware — consolidates pods onto fewer nodes, terminates underutilized ones. Disruption budgets for graceful consolidation.

Spot strategy: NodePool with multiple instance families and sizes. Spot interruption handling: Karpenter detects 2-minute warning via EC2 metadata, cordons and drains gracefully. Mix spot and on-demand — spot for stateless workloads, on-demand for stateful/databases. With 10+ instance types in pool, interruption rate drops to near zero.

How do Kubernetes service account tokens work with IRSA (AWS) and Workload Identity (GCP) for secure cloud API access?
AWS IRSAGCP WI
+

Both IRSA and GCP Workload Identity use OIDC token projection to give pods cloud IAM identities without static credentials.

Flow (IRSA): Pod's ServiceAccount has an annotation linking it to an IAM Role. K8s projects an OIDC JWT token into the pod filesystem. AWS SDK exchanges this token with STS AssumeRoleWithWebIdentity for temporary AWS credentials. The OIDC issuer is the EKS cluster — AWS trusts it because you configured the IAM OIDC provider.

Fine-grained access: Each microservice gets its own ServiceAccount and IAM Role with minimum permissions. A payment service's role has S3 access to its bucket only. No static credentials anywhere — credentials are ephemeral and auto-rotated by STS.

💡 This eliminates the #1 cloud security risk — static AWS credentials in environment variables or code. Zero secrets required in your pipeline when combined with GitHub OIDC for CI/CD.

K8s Gateway API vs Ingress — why is Gateway API the future and what's the migration path?
Kubernetes networking
+

Ingress limitations: Single resource type for all traffic routing. Complex features (canary, header-based routing, TCP/UDP) require annotations that differ per ingress controller — no portability. All configuration owned by cluster admins (no delegation to app teams).

Gateway API advantages: Role-based model: GatewayClass (infra admin), Gateway (cluster operator), HTTPRoute (app developer). App teams own their routing rules without admin access. Standard API across all implementations (NGINX, Traefik, Envoy Gateway, Istio). Supports HTTP, TCP, TLS, gRPC natively without annotations. Canary and traffic splitting built into HTTPRoute spec.

Migration: Most controllers now support both — run in parallel. Migrate service by service to HTTPRoute, keeping Ingress as fallback. No downtime migration is straightforward since they can coexist. Full migration typically over 1-2 quarters for large clusters.

A pod can't reach an external service. Walk through your systematic Kubernetes networking troubleshooting process.
CKA scenarioDebug
+

Layer by layer:

  • Pod level: kubectl exec <pod> -- curl -v <target> — can the pod make the request at all?
  • DNS: kubectl exec <pod> -- nslookup <hostname> — resolves correctly? Check CoreDNS pods are running, check /etc/resolv.conf in the pod.
  • NetworkPolicy: Check if any NetworkPolicy blocks egress from this namespace/pod. A deny-all egress policy without an allow rule is a common issue.
  • NAT/routing: Can pod reach its node's IP? Can it reach cluster CIDR? Check if NAT Gateway/internet gateway configured for the node's subnet (for external connectivity).
  • Service (if applicable): kubectl get endpoints <service> — are backend pods registered? Check selector matches pod labels.
  • CNI: Check CNI plugin logs. Cilium: cilium connectivity test. Calico: calicoctl node status.

How does Kubernetes pod priority and preemption work, and when would you use PriorityClasses in production?
K8s scheduling
+

PriorityClasses assign integer priority values to pods. Higher-priority pods can preempt (evict) lower-priority pods on a node if resources are insufficient.

Production use cases:

  • Critical infrastructure: Monitoring, logging agents — PriorityClass: system-cluster-critical or custom high priority. Never evicted even under pressure.
  • Tiered workloads: Production services > batch jobs > dev workloads. When cluster is full, batch jobs get preempted before production services.
  • Spot handling: Spot reclaim triggers drain. PriorityClass ensures stateful services migrate before stateless ones.

Caution: Preemption disrupts the preempted pods. Set thoughtful PriorityClasses — not everything should be high priority (if everything is high priority, nothing is). Combine with PodDisruptionBudgets to control preemption impact.

How do you reliably run batch workloads in Kubernetes, and what patterns prevent zombie jobs and missed schedules?
Batch processing
+

CronJob gotchas: startingDeadlineSeconds — if the CronJob controller is down for longer than this, the job is considered missed. Set appropriately for your SLA. concurrencyPolicy: Forbid prevents overlapping runs for non-idempotent jobs. successfulJobsHistoryLimit and failedJobsHistoryLimit control cleanup — don't let failed jobs pile up.

Zombie jobs: Set activeDeadlineSeconds on Job to kill hung jobs automatically. Set backoffLimit to prevent infinite retry loops. Use TTL controller (ttlSecondsAfterFinished) for automatic cleanup.

Production patterns: Use Argo Workflows or Temporal for complex DAG jobs where CronJob is insufficient. KEDA ScaledJob for event-triggered batch (process SQS queue) — spins up pods per message, terminates when queue empty. This is more efficient than long-running batch consumers.

How do validating and mutating admission webhooks work, and what are the risks of poorly configured webhooks?
K8s internals
+

When a resource is created/updated, the API server calls configured admission webhooks. Mutating webhooks can modify the resource (add labels, inject sidecars, set defaults). Validating webhooks can allow or deny the request. Mutating runs first, then validating.

Risk: Webhook availability = cluster availability. If your webhook server is down and configured as failurePolicy: Fail, ALL pod creation fails. This has caused major outages. Best practices:

  • Set failurePolicy: Ignore for non-critical webhooks — fail open, not closed
  • Exclude kube-system namespace from webhook scope — ensures control plane components always schedule
  • Run webhook servers with HA (2+ replicas, PodDisruptionBudget)
  • Set aggressive timeouts (5s) to fail fast rather than blocking API server
  • Test webhook behavior during upgrades and node failures

How do you perform zero-downtime database schema migrations in a system that cannot tolerate read or write downtime?
StripeShopify
+

Expand-Contract pattern (the only reliable approach):

  • Expand: Add new column/table alongside old one. Deploy code that writes to BOTH old and new. Back-fill old data to new column in small batches (avoid locking full table).
  • Migrate reads: Deploy code that reads from new column, falls back to old if null. Verify new column has 100% data.
  • Contract: Remove writes to old column. Verify no reads. Drop old column in a later release.

Renaming a column takes 3+ deployments this way — never do it in a single deployment. Tools like gh-ost (GitHub) perform online schema changes on MySQL without locking. PgHerd / pg_repack for Postgres. Never run ALTER TABLE that acquires an exclusive lock on a production table with millions of rows.

What is consumer-driven contract testing (Pact), and why does it replace integration test environments for microservices?
ITVREA Group
+

Problem: Integration test environments are expensive, flaky, and slow. You need all services running simultaneously. When service A changes its API, service B (the consumer) breaks in production — you only discover this during integration testing or in prod.

Contract testing: Consumer defines the contract (what it expects from the provider API). Contract stored in Pact Broker. Provider verifies it can satisfy the contract in its own CI without consumer being deployed. Breaking change to provider API fails provider's CI before deployment — instant feedback, no shared environment needed.

Result: Each service deploys independently and confidently. "Can I deploy?" check verifies all contracts pass. Enables independent service deployments at high frequency. Particularly powerful for GraphQL schemas where breaking changes are easy to accidentally introduce.

How do you implement automated rollback in a deployment pipeline triggered by production metrics degradation?
Argo RolloutsNetflix
+

Argo Rollouts analysis templates: Define metric queries (Prometheus, Datadog, Wavefront) as AnalysisTemplate. Rollout pauses at each canary step, evaluates metrics against thresholds. If error rate > 1% or p99 latency > 500ms → automatically abort, rollback to previous version. No human needed.

How do you run a blameless postmortem that actually improves systems rather than just documenting what went wrong?
Google SREEtsy
+

Why blameless matters: If engineers fear blame, they hide information. You learn less from each incident. You lose good engineers who feel scapegoated. Blameless postmortems assume humans were doing their best given the information they had — the system design enabled the failure.

Structure for actionable postmortems: Timeline (what happened, when, who noticed). Contributing factors (not root cause — complex systems have multiple contributing factors, not a single root cause). Impact (users affected, revenue loss, SLO impact). Action items with owners and due dates — this is the most important part.

Making it actually work: Action items must be tracked in a system. Follow up in 2 weeks — were they done? Track repeat incidents — the same postmortem action item failing is a signal the organization isn't investing in reliability. Senior leadership must publicly celebrate learning from incidents, not just celebrating uptime.

How do you build a capacity planning process for a system with unpredictable viral growth spikes?
TwitterReddit
+

Two-track approach: Predictable growth (long-term trend) uses forecasting models (linear regression on traffic growth). Unpredictable spikes need headroom + fast scaling.

Headroom strategy: Always run at 50-70% capacity at typical load, not 90%. This gives headroom for 2x spikes without degradation. Overcapacity has a cost — weigh it against the risk of degradation. For ad-supported services, degradation during viral events means lost revenue that exceeds overprovisioning cost.

Automated response: KEDA or HPA triggers at lower thresholds. Pre-provision warm pool of spare capacity (EC2 Auto Scaling warm pool). Load shed gracefully — queue non-critical requests before failing them. Publish rate limits to API consumers so clients can back off intelligently. Game days: simulate 10x traffic, measure where bottlenecks appear, fix before they hit production.

The SRE team spends 70% of their time on operational toil. How do you identify and systematically eliminate it?
Google SRE model
+

Toil definition: Manual, repetitive, automatable work that scales with service growth and produces no lasting value. Rotating log files, responding to false-positive alerts, approving routine access requests, restarting services on OOM, manually running deployment scripts.

Measurement: Track toil hours with time-based tagging in incident tickets and on-call logs. Categorize by type. Visualize weekly toil breakdown by category. Makes the problem visible and measurable.

Elimination playbook: Prioritize by frequency × time-cost. Top 3 toil sources get dedicated automation sprints. Auto-remediation runbooks: common alerts trigger automated remediation (restart pod, scale up, flush cache) before human is paged. Self-service tooling for access provisioning, certificate rotation, deploy approvals. Google's SRE model: max 50% toil allowed — rest must be project/engineering work. Enforce the 50% ceiling by refusing to add to on-call rotation without eliminating corresponding toil.

How do circuit breakers, bulkheads, and retries work together to prevent cascade failures in microservices?

+

Explain Mixture of Experts (MoE) architecture — why does Mixtral use it, and what are the engineering challenges of deploying MoE models?

Why does extending an LLM's context window after training fail without special techniques, and how do RoPE and YaRN solve this?
LLM internals
+

Models trained with positional encodings up to position N have never seen positions beyond N. When you pass a longer sequence, the model encounters out-of-distribution positional values and performance degrades significantly.

RoPE (Rotary Position Embedding): Encodes position as a rotation in embedding space. Relative positions are represented by rotation differences. Position interpolation (PI): scale position indices down proportionally to fit longer context into trained range. E.g., scale 128K context to fit in 4K training range by multiplying all positions by 4K/128K. Requires fine-tuning on long-context data to recover performance.

YaRN: Improves on PI with a smarter interpolation — different scaling for different frequency components of RoPE. Fewer long-context fine-tuning steps required. Llama models extended with YaRN achieve near-original performance at 4-8x original context length.

How does FlashAttention make long-context training practical, and what does it change about memory usage?
StanfordGPU optimization
+

Standard attention memory problem: The attention score matrix is N×N (sequence length squared). For a 32K token sequence, the N×N matrix is 1 billion floats — 4GB just for one layer's attention matrix. Training with 128K context would require terabytes of GPU memory just for attention.

FlashAttention insight: Don't materialize the full N×N matrix in GPU HBM (slow memory). Compute attention in tiles that fit in fast SRAM, using online softmax normalization. The result is mathematically identical to standard attention, but never writes the full N×N matrix to HBM.

Memory reduction: O(N²) memory → O(N) memory. Allows training on sequences 8-16x longer on the same GPU. Also significantly faster — HBM bandwidth is the bottleneck for standard attention; FlashAttention reduces HBM reads/writes by 10x. FlashAttention 2/3 are standard in all modern LLM training stacks (transformers library, vLLM).

How does speculative decoding work and what determines how much speedup you actually get in practice?
LLM serving
+

Mechanism: A small, fast "draft" model generates K candidate tokens. The large "target" model verifies all K tokens in a single parallel forward pass (much faster than sequential generation). Accept tokens where draft and target agree; reject and regenerate from the first disagreement. Accepted tokens are correct by construction — no quality degradation.

Speedup determination: Key metric is token acceptance rate. If the draft model agrees with the target 80% of the time, you accept ~4 tokens per target forward pass on average → ~3-4x speedup. If acceptance rate is 50%, speedup is minimal. Acceptance rate depends on how well the draft model distribution matches the target — domain-specific draft models work much better than generic ones. Works best for repetitive/predictable text; worst for creative, high-entropy generation.

In practice: 2-3x speedup is common for code completion (predictable). 1.3-1.7x for conversational (less predictable). Medusa and other self-speculative approaches use the model's own layers as the draft — no separate model needed, simpler to deploy.

Explain KV caching in LLM inference — what is prefix caching and how does it dramatically reduce costs for certain applications?

+

During autoregressive generation, the model computes key-value (KV) tensors for each token in the context. These are cached so subsequent tokens don't recompute previous tokens' attention. KV cache grows linearly with context length — a major memory consumer in serving.

Prefix caching: If many requests share the same prefix (same system prompt, same few-shot examples, same document being queried), the KV cache for that prefix can be computed once and shared across all requests. Second and subsequent requests with the same prefix skip recomputation entirely.

Cost impact: For RAG applications where the same documents are retrieved repeatedly, or chatbots with long system prompts, prefix cache hit rates of 80-90% are achievable. Anthropic's API charges 10% of normal price for cached input tokens (prompt caching). On 10K token system prompt × 1M daily requests, prefix caching reduces input token cost by 90% for those tokens.

What is Anthropic's Contextual Retrieval technique and how does it improve retrieval accuracy?

+

How does multi-query retrieval work and why does it improve recall for ambiguous queries?
RAG techniques
+

Multi-query retrieval uses an LLM to generate N diverse paraphrases of the original query, then retrieves documents for each paraphrase. Results are merged (deduplicated using Reciprocal Rank Fusion). The intuition: different phrasings of the same question retrieve different relevant documents because they match different writing styles in the corpus.

Example: "What are the side effects of aspirin?" → also generates "adverse reactions to aspirin," "aspirin risks and warnings," "salicylate adverse effects." Each retrieves different but relevant passages. Merge gives higher recall than any single query alone.

Trade-offs: 3-5x more LLM calls for query generation. 3-5x more vector DB queries. Higher latency if sequential; manageable if parallel. Worth it for high-stakes QA (medical, legal) where missing relevant info is costly. For latency-sensitive chat, consider single-query with HyDE instead.

Explain the "small-to-big" retrieval pattern — why retrieve small chunks but send larger context to the LLM?
RAG architecture
+

The tension: Small chunks (64-128 tokens) give precise retrieval — the embedding represents a specific idea, not a diluted average of many ideas. But small chunks lack context — a sentence may need the surrounding paragraph to be meaningful to the LLM.

Small-to-big pattern: Index small child chunks (128 tokens) for retrieval. Each child chunk points to a parent chunk (512-1024 tokens). When a child chunk is retrieved, swap it for the parent before sending to LLM. Best of both: precise retrieval matching + contextual generation input.

Sentence window variant: Retrieve individual sentences, send a sliding window of ±3 sentences around the matched sentence. LlamaIndex implements both patterns. Reduces LLM hallucination from missing context while maintaining retrieval precision. Particularly effective for technical documentation where specific sentences matter but context is critical for correct interpretation.

How do you implement task decomposition in an agent that needs to complete complex, multi-step goals autonomously?

+

Plan-and-execute pattern: Separate planning from execution. Planning LLM (larger, smarter) creates an explicit step-by-step plan for the goal. Execution LLM (can be smaller) executes each step. What is reflexion / self-reflection in agents and when does having the agent critique its own work improve quality?

+

Reflexion (Shinn et al.) has the agent evaluate its own response against explicit criteria, identify shortcomings, and iterate. The agent writes code → runs tests → sees failures → reflects on why it failed → rewrites → repeat until tests pass or N iterations.

When it works: Tasks with clear, verifiable success criteria (code passes tests, document meets specified length/format, fact-checked answer). The critique prompt must be specific — "evaluate if this code handles edge cases X and Y" outperforms "is this good?"

Caveat: If the model can't generate a correct response, it also can't critique it accurately. Self-reflection doesn't fix fundamental model capability gaps — it helps polish outputs the model is capable of producing.

Explain tensor parallelism vs. pipeline parallelism vs. data parallelism for distributed LLM training — when would you use each?
Distributed training
+

Data parallelism: Each GPU has a full model copy. Different data batches processed in parallel. Gradients averaged across GPUs. Easy to implement, scales to many GPUs for batch parallelism. Requires model to fit in single GPU memory — doesn't help for large models.

Tensor parallelism: Split individual model layers across GPUs (matrix rows/columns divided). All GPUs collaborate on every forward pass — requires fast interconnect (NVLink). Good for reducing per-GPU memory. Communication intensive — best within a single node (8 GPUs connected via NVLink). Megatron-LM implements this.

Pipeline parallelism: Split model layers into stages across GPUs. Stage 1 processes layers 1-12, stage 2 processes layers 13-24. Micro-batching hides stage bubbles (idle time). Good for multi-node training where interconnect is slower. Less communication than tensor parallel.

In practice: Large model training uses 3D parallelism: tensor parallel within node, pipeline parallel across nodes, data parallel across pipeline replicas. DeepSpeed ZeRO partitions optimizer state for memory efficiency on top of this.

How do you maximize GPU utilization in an LLM inference serving system handling thousands of concurrent requests?

Cost reduction → "frees $Y/month for product investment." Reliability improvement → "reduces customer churn by Z% based on our support data showing reliability is the #1 complaint."

Risk communication: Be honest about trade-offs. "This migration will require 2 months of engineering time and there's a 20% chance we'll need an additional month." Stakeholders respect honesty; they distrust engineers who present only upside. Present multiple options with different risk/reward profiles — this shows you've thought deeply and gives stakeholders agency.
